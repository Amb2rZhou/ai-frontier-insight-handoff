## LLM-Wiki 写入机制（详细）

每次采集到新信号时，LLM 同时执行：
- 摘要写入日报
- 关联已有 wiki 页面（如"Meta Orion"页面），更新产品时间线/技术演进
- 必要时创建新 wiki 页面
- 维护交叉引用（`[[wiki-link]]`）

**页面类型**：
- 产品线（如 Meta Orion → 1 代/2 代/3 代，每代关键变化）
- 技术方向（如 MCP 协议 → 提出/演进/采用情况）
- 公司动态（如 OpenAI → 产品发布时间线）
- 趋势（如 Agent 可靠性 → 各周数据点串联）

**查询机制**：用户提问时，先查 wiki 结构化知识，再补充检索原始信号，综合回答。

## 交互场景（完整）

| 场景 | 触发 | 行为 |
|------|------|------|
| 提问 | @bot XX方向最近有什么进展？ | 查 wiki + 近期信号，综合回答 |
| 追问 | @bot 今天日报第3条展开讲讲 | 拉原始数据源，深入分析 |
| 脉络 | @bot XX产品的发展历程 | 从 wiki 产品页拉时间线 |
| 待办 | @bot 跟踪XX，有更新提醒我 | 写入 follow-up 列表，后续命中时主动推送 |
| 对比 | @bot A和B方案对比 | 从 wiki 拉两个条目做结构化对比 |

## 技术栈

- **运行环境**：内部服务器（云端）
- **LLM**：内部模型 API 或 agent 调用
- **框架**：Python（FastAPI 或类似），常驻服务
- **消息接口**：Hi API（回调 + 主动推送）
- **知识存储**：wiki 页面（markdown），向量索引辅助检索
- **数据源 API**：X API（按量付费，~$66/月）、RSS、ArXiv、GitHub、HuggingFace
- **Wiki 可视化**：Quartz 静态站点，部署到内部域名

## 与 v1 的关系

- v1 代码可直接复用：采集层、分析层、wiki 更新、格式化、prompt 模板
- v2 新增部分：X API collector、交互层（Hi API 回调）、wiki 自动建页、向量索引
